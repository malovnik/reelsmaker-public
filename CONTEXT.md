# videomaker — контекст для планировщика

## Цель проекта
Локальное десктоп-приложение (веб-интерфейс на localhost) для автоматической нарезки длинных видео на короткие вертикальные клипы (рилсы/шортсы 9:16) с использованием мультипроходного LLM-анализа транскрипта, умной склейки кусков из разных частей исходника, B-roll генерации и финального экспорта в оптимизированном виде.

## Железо (жёсткое ограничение)
- MacBook Pro, Apple Silicon M5, 24 GB RAM.
- Весь стек должен быть оптимизирован под ARM + Metal/MPS/ANE.
- Если есть выбор между универсальным решением и нативным под Apple Silicon — брать нативное.

## Первый этап (MVP scope)
- Вход: один видеофайл (любая длина, обычно 20–90 минут).
- Выход: набор коротких клипов 9:16 (реелс/шортс), 30 fps, H.265 (HEVC), ≥15 000 kbps битрейт при оптимальном сжатии.
- Вертикаль 9:16 — приоритет, но архитектура должна позволить 16:9, 1:1, 4:5 без рефакторинга (соотношение — параметр пайплайна).

## Функциональные узлы пайплайна
1. **Загрузка видео** через веб-UI (локально, без upload в облако).
2. **Транскрибация с таймкодами на уровне слов.**
   - Основной путь: Deepgram API (nova-3 / nova-2), если получаем word-level timestamps.
   - Альтернативный путь: локальный Whisper (`whisper.cpp`/`faster-whisper`/`MLX-Whisper`) — выбор должен быть конфигурируемым.
   - Обязательно: word-level timestamps + confidence, чтобы потом точно резать по границам слов без обрубленных звуков.
3. **Удаление тишины и филлеров.**
   - Тишина: по энергии/RMS + по отсутствию слов в транскрипте.
   - Филлеры русского языка: «эээ», «ммм», «ну», «как бы», «типа», «короче», «бляя», «пыканье», прочищение горла и т.п. — конфигурируемый список + ML-классификатор на границе.
4. **Мультипроходный LLM-анализ транскрипта** (вдохновляться структурой `universal-rag/packages/backend/app/services/card_creator.py` и `prompts.py` — там паттерн каскадных промптов).
   - Проход 1: буквальные смысловые единицы (где человек прямо высказал тезис).
   - Проход 2: неявные смыслы, углы подачи, эмоциональные пики.
   - Проход 3: склейки из разных фрагментов видео — LLM предлагает, какие 2–5 коротких фраз из разных мест можно смонтировать в один смысловой рилс.
   - Выход: JSON со сценариями рилсов, где каждый рилс = список сегментов `{start, end, source_timecode, reasoning}`.
   - Модель по умолчанию: **Gemini 2.5 Flash**, с возможностью переключения на Gemini 2.5 Pro / Claude Sonnet / Opus / GPT-5 через конфиг.
5. **Резка и склейка** по таймкодам из LLM-плана (ffmpeg/PyAV), без перекодирования на промежуточных шагах где возможно (stream copy), финальный рендер — единый pass.
6. **Переходы зумом** (Ken Burns / punch-in) между склеенными фрагментами.
7. **Цветокоррекция** (фильтры/LUT) — конфигурируемые пресеты.
8. **Субтитры**: генерация из транскрипта + LLM-проход на исправление опечаток/склонений/пунктуации **без сдвига таймкодов** (каждое слово остаётся привязанным к своему исходному таймингу). Вшивание burn-in или SRT/ASS.
9. **B-roll / графические вставки**: генерация картинок через **Google Nano Banana (Gemini image generation)** в разрешении 1024×... (1K). Архитектура должна позволить подменить на DALL-E / GPT-image / Midjourney / Flux.
10. **Всплывающие плашки** «подписаться», CTA — конфигурируемые шаблоны (позиция, длительность, анимация).
11. **Экспорт**: H.265, 30 fps, ≥15 Mbps, правильный VBR/CRF для минимального размера при целевом качестве. Отдельно hardware-encoding через VideoToolbox (Apple Silicon — обязательно, иначе рендер будет часами).

## Веб-интерфейс (локальный)
- Next.js 15 / React 19 или Python-based (FastAPI + minimal React) — **оценить оба варианта в плане, выбрать обоснованно**.
- Должен позволять:
  - Загружать видео, видеть прогресс пайплайна по стадиям (SSE/WebSocket).
  - Редактировать промпты для каждого прохода LLM (cutter, B-roll, subtitle fixer).
  - Менять настройки: модель LLM, провайдер транскрипции, пресет цвета, список филлеров, целевая длина клипов, формат выхода.
  - Превью сгенерированных клипов до финального экспорта.
  - Ручная правка таймкодов перед рендером (drag-and-drop на таймлайне).

## Стек-кандидаты (для оценки в плане)
- Python (FastAPI) + PyAV/ffmpeg-python — удобно для ML/LLM.
- Node.js + fluent-ffmpeg — удобно для UI, но ML-обвязка сложнее.
- Rust/Tauri для десктоп-обёртки — оверкилл на MVP, но заложить возможность миграции.
- Orchestration: один процесс vs worker queue (celery/arq/redis) — для локального приложения скорее всего хватит fastapi + background tasks, но проверить устойчивость на длинных видео.

## Нефункциональные требования
- **Производительность**: на 60-мин видео полный пайплайн должен укладываться в разумное время (целевая метрика — проработать в плане, но ориентир: < 1.5× realtime на M5).
- **Устойчивость**: длинные операции (Whisper, рендер) должны быть прерываемыми и возобновляемыми, с промежуточными артефактами на диске.
- **NO MOCKS, NO STUBS** (жёсткое правило): все узлы пайплайна с первой версии — рабочие с реальными API/моделями, никаких `// TODO`, никаких заглушек.
- **Безопасность API-ключей**: Deepgram, Gemini API keys — через .env, никогда в коде/UI-state.

## Открытые вопросы для плана (на которые планировщик должен дать ответы с обоснованием)
1. Whisper-вариант для M5: MLX-Whisper (Apple) vs faster-whisper (CTranslate2) vs whisper.cpp — что даёт лучший tradeoff точность/скорость с word timestamps?
2. Формат хранения промежуточных артефактов между стадиями (JSON + отдельные медиафайлы vs SQLite + BLOB vs просто папка с файлами и манифестом)?
3. Ken Burns / zoom-transitions: ffmpeg-фильтры vs PyAV composer vs MoviePy — что даёт нужный FPS + качество?
4. Burn-in субтитров: libass через ffmpeg vs рендер через Skia/Pillow на кадры — где контроль над стилем лучше?
5. Hardware-encoding HEVC через VideoToolbox: параметры для ≥15 Mbps при минимальном размере (VBR vs ABR vs CRF 23 + max-rate)?
6. Оркестрация LLM-проходов: последовательно vs параллельно (для рилсов-из-разных-мест — нужен граф зависимостей).
7. Нано-банана через Gemini API: формат вызова, поддержка 1024px, rate limits, batch generation.
8. Стратегия матчинга B-roll к смысловому блоку: keyword-based vs semantic search по embeddings.
