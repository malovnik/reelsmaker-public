## Суть этапа

На входе — карта аудита Этапа 00. На выходе — зафиксированные архитектурные решения, каждое с обоснованием (Sequential Thinking FOR → AGAINST → VERDICT + Context7 документация). Решения записываются как ADR (Architectural Decision Records) в `docs/adr/`.

Этап закрывает вопросы: какой стек для фронта, где хранить состояние проекта, как устроено автосохранение, какой путь рендера на M5, как работает система тем. Без этих решений этапы 02–09 рассыплются.

**Режим работы:** Sequential. Каждый чанк = один ADR.

## Подэтапы (REFACTR-07..REFACTR-12)

- **REFACTR-07** — ADR: Frontend-стек (Vite + React + TanStack vs альтернативы) ⬜
- **REFACTR-08** — ADR: Хранение данных (SQLite + project.json, миграции) ⬜
- **REFACTR-09** — ADR: Автосохранение (debounce, конфликт-резолюция) ⬜
- **REFACTR-10** — ADR: Видеодвижок на M5 Pro (VideoToolbox + fallback) ⬜
- **REFACTR-11** — ADR: Темы (CSS variables + persist двухсторонний) ⬜
- **REFACTR-12** — Итоговая архитектурная диаграмма (C4 level 1-2) ⬜

## Вход

- 7 документов аудита из Этапа 00.
- Требования владельца (перечислены в `README.md` RM-CHUNKS).
- Context7 для верификации API/возможностей библиотек.

## Выход

- 5 ADR в `docs/adr/NNNN-*.md` по шаблону MADR.
- 1 архитектурная диаграмма в `docs/architecture/c4-overview.md` + `.svg` при наличии Draw.io/Excalidraw.

## Формат ADR

```markdown
# ADR-NNNN: <Решение>

## Статус
Принят | Заменён | Отменён — дата

## Контекст
<2-4 абзаца: какая проблема, какие силы действуют>

## Рассматриваемые варианты
### Вариант A
+ плюсы  — минусы
### Вариант B
+ плюсы  — минусы
### Вариант C
+ плюсы  — минусы

## Решение
<Выбранный вариант, ключевой критерий>

## Последствия
- Позитивные
- Негативные
- Нейтральные

## Верификация
<Как проверяем, что решение правильное: бенчмарк, smoke-тест, метрика>
```
