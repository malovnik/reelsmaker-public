# mac-2-diagnostics — Reelibra: диагностика старта и чистка висяков (macOS)

Recon-документ для launcher'а Reelibra на macOS. Описывает: схему диагностики
окружения, чистку зависших процессов/портов, детект "жёстко закрыли в прошлый
раз" и UX-поток. Источники: `run.sh`, `apps/backend/src/videomaker/main.py`,
`services/jobs.py`, `services/proxy.py`, `services/publer/worker.py`.

---

## 0. Топология процесса

| Компонент | Команда | Порт | cwd |
|---|---|---|---|
| Backend | `uv run uvicorn videomaker.main:app --reload --reload-dir src` | `8000` (`APP_HOST=127.0.0.1`) | `apps/backend` |
| Frontend | `pnpm dev` → `vite` | `3000` (`strictPort: true`) | `apps/frontend` |

Vite жёстко требует 3000 (`strictPort`) — занятый порт = **fatal**, не сдвиг
на 3001. Это делает освобождение порта 3000 обязательным шагом, а не желательным.

Дерево процессов на одном запуске (то, что надо ловить при чистке):
```
run.sh
├── uv run uvicorn ...        → python (uvicorn reloader)
│   └── python (uvicorn worker, реальный app + PublerWorker + ffmpeg-дети)
│       └── ffmpeg ... data/artifacts/...   (рендеры, могут пережить родителя)
└── pnpm dev → node (vite)
    └── esbuild --service ...  (дочерний процесс Vite)
```

Что переживает крах родителя (сироты, цель чистки):
- uvicorn worker (двойной процесс из-за `--reload`: reloader + worker);
- `ffmpeg`-рендеры (spawn'ятся пайплайном, не привязаны к TTY);
- `esbuild --service` (детач от node).

## 1. Схема диагностики окружения (каждый запуск)

Принцип: идемпотентно. Есть и валидно → "ОК" за <1с. Нет → доустановка с
прогрессом. Порядок — от блокеров к мягким зависимостям.

### 1.1 Бинарные зависимости (блокеры)

| Проверка | Команда детекта | Ожидание | Если нет |
|---|---|---|---|
| uv | `command -v uv` | присутствует | `curl -LsSf https://astral.sh/uv/install.sh \| sh` |
| pnpm | `command -v pnpm` | присутствует | `npm install -g pnpm` (нужен Node) |
| Node | `command -v node && node -v` | major ≥ 20 (Vite 5+) | `brew install node` или nvm |
| ffmpeg | `command -v ffmpeg` | присутствует | `brew install ffmpeg` |
| Python | через uv (`requires-python >=3.12,<3.13`) | 3.12.x | `uv python install 3.12` |

run.sh уже падает с человекочитаемым сообщением при отсутствии uv/pnpm/ffmpeg
(строки 26-28). Launcher должен **расширить**: добавить Node-версию (pnpm без
Node бесполезен) и явную проверку Python 3.12 (uv сам подтянет, но прогресс надо
показать — установка интерпретатора занимает время).

### 1.2 Целостность окружений

| Артефакт | Проверка наличия | Проверка целостности | Действие при провале |
|---|---|---|---|
| Backend `.venv` | `apps/backend/.venv/bin/python` существует | `uv sync` (сверяет с `uv.lock`) | `uv sync` — пересоберёт |
| Frontend `node_modules` | `apps/frontend/node_modules/.bin/vite` | `pnpm install` (сверяет lockfile) | `pnpm install` |
| `.env` | файл существует | содержит `GEMINI_API_KEY` непустой | копия из `.env.example` + предупреждение |
| `data/` дерево | каталоги существуют | — | `mkdir -p data/{uploads,artifacts,logs}` (+ proxies/thumbnails/transcripts/face_cache/vision_cache/models/post_production_assets) |

`uv sync --quiet` и `pnpm install --silent` сами по себе идемпотентны и быстры
при неизменном lock — это и есть проверка целостности. Доустановка = просто их
запуск без `--quiet`, чтобы пользователь видел прогресс скачивания.

`.env`: run.sh копирует из примера, но **не проверяет ключи**. Launcher должен
предупредить, если `GEMINI_API_KEY` пуст (без него пайплайн не работает) —
но не блокировать (UI поднимется, юзер впишет ключ в `/settings`).

### 1.3 Схема (псевдокод диагностики)

```
для каждого шага в [uv, pnpm, node, ffmpeg, python312,
                     backend_venv, frontend_modules, env, data_dirs]:
    печать "Проверяю <шаг>… "
    если ok:           печать "ОК"
    иначе:             печать "не найдено → ставлю…"; установить с прогрессом; пере-проверить
    если всё ещё нет:  печать понятной ошибки; стоп (для блокеров) | warn (для .env)
```

## 2. Чистка висяков и портов (macOS)

Двухканальный поиск — по **порту** (надёжно для слушающих сокетов) и по
**паттерну команды** (ловит сирот-ffmpeg/esbuild, которые порт не держат).
run.sh уже делает базовую версию (`preflight_kill` + `preflight_free_port`);
ниже — полная карта паттернов и порядок graceful→hard.

### 2.1 Поиск по портам (lsof)

```bash
# PID'ы, слушающие порт (только LISTEN, не клиентские коннекты)
lsof -nP -iTCP:8000 -sTCP:LISTEN -t
lsof -nP -iTCP:3000 -sTCP:LISTEN -t
```
- `-t` — только PID (чище чем `-Fp`+sed из run.sh, тот же результат).
- `-sTCP:LISTEN` критично: иначе поймаем браузерные вкладки, висящие на 3000.

### 2.2 Поиск по паттерну (pgrep -f)

| Паттерн | Что ловит |
|---|---|
| `uvicorn videomaker.main` | uvicorn worker (реальный app) |
| `uv run uvicorn` | uv-обёртка/reloader |
| `node .*vite` | Vite dev-сервер |
| `esbuild --service` | дочерний esbuild Vite |
| `pnpm.*dev` | pnpm-обёртка |
| `ffmpeg.*data/artifacts` | residual рендеры Reelibra |
| `ffmpeg.*data/proxies` | **(добавить)** residual proxy-транскоды |

Якорить по `data/artifacts`/`data/proxies` обязательно — нельзя `pkill ffmpeg`
вслепую, юзер может рендерить что-то своё в другом приложении.

### 2.3 Порядок завершения: graceful → SIGKILL

run.sh бьёт сразу `kill -9` в preflight. Корректнее (особенно для уже работающей
сессии через trap) — дать шанс на graceful shutdown, чтобы отработал lifespan
`finally` (flush_all, dispose_engine, publer_worker.stop):

```
1. SIGTERM найденным PID'ам
2. ждать до 5с в until-цикле, проверяя что процесс исчез
3. кто выжил → SIGKILL
4. re-check портов lsof: пусто → ОК
```

SIGKILL backend пропускает lifespan-`finally` — но это безопасно: stale-jobs
и зависшие proxy-locks подхватываются на следующем старте (см. §3). То есть
hard-kill допустим как fallback, данные не теряются.

### 2.4 Удаление мусорных файлов (без потери БД/данных)

**Защищать (НЕ трогать):** `data/videomaker.db`, `data/uploads/`,
`data/artifacts/` (готовые рилсы), `data/transcripts/`, `data/models/`,
`data/post_production_assets/`.

**Чистить (безопасно — это незавершённый промежуточный мусор):**

| Паттерн | Где | Почему безопасно |
|---|---|---|
| `*.partial` | `data/proxies/`, везде | прерванная atomic-запись ffmpeg, не финальный файл |
| `*.lock` | `data/proxies/`, `vision_cache/.locks/` | orphan-локи от убитых процессов |
| `*.tmp` | `data/proxies/`, `transcripts/`, `face_cache/`, `_pending/` | незавершённые atomic-write temp |
| `__pycache__/` | `apps/backend/src` | stale bytecode (run.sh уже чистит) |

Важно про `.lock`: код (`proxy.py:_acquire_lock`) сам чистит orphan-локи по
`mtime > lock_timeout_sec` (1800с). Launcher может удалить их сразу при старте
(процесс точно мёртв после kill) — но это **опционально**, код самовосстановится.
Удалять только `.lock` файлы старше нескольких секунд (не задеть свежий лок
параллельного запуска — хотя при single-user launcher это не актуально).

`_pending/*.tmp` (asset_store) и `*.partial` — всегда безопасно удалять: по
определению это незакоммиченные загрузки/транскоды.

## 3. Детект "жёстко закрыли в прошлый раз" (stale state)

Три независимых индикатора аварийного завершения. Любой → выполнить чистку.

### 3.1 Занятый порт без нашего процесса
`lsof -iTCP:8000 -sTCP:LISTEN` возвращает PID → прошлый backend жив/завис.
Это самый частый случай "вкладка/процесс остались".

### 3.2 Running-jobs в БД (уже решено в коде)
`reset_stale_running_jobs` (`jobs.py:908`) при каждом старте lifespan переводит
**все** `Job.status == running` → `error` с пометкой
`"interrupted by application restart"`. Аналогично lifespan (`main.py:79-89`)
сбрасывает Publer `ScheduleAssignmentRow` из `uploading` → `queued`.

Вывод для launcher: **БД-уровень самовосстанавливается** — отдельная чистка
running-jobs не нужна, достаточно дать backend стартовать. Launcher только
**информирует** пользователя постфактум (читает лог `stale_jobs_reset_on_startup`
или последний health-ответ), не вмешиваясь в БД сам. Не дублировать логику —
писать в SQLite из shell-скрипта запрещено (риск повредить незакрытый WAL).

### 3.3 Orphan proxy-locks
`data/proxies/*.lock` присутствуют после краша — см. §2.4. Код чистит по таймауту,
launcher может ускорить.

### 3.4 (Опционально) PID-файл
Текущий проект **PID-файла не пишет**. Если launcher хочет надёжный
self-detect — писать `data/.reelibra.pid` со своим PID при старте, на старте
читать: если PID жив и это наш процесс-дерево → "уже запущено" (не плодить
второй экземпляр); если PID мёртв, а файл есть → прошлый запуск рухнул, чистить.
Это единственное реальное **дополнение** к существующему коду, остальное уже есть.

Граф решения:
```
старт launcher
 ├─ порт 8000/3000 занят? ──да──┐
 ├─ data/proxies/*.lock есть? ─да┤→ "обнаружен незавершённый прошлый запуск"
 ├─ .reilibra.pid мёртвый? ───да─┘   → §2 чистка (graceful→kill, удалить мусор)
 └─ всё чисто → сразу к запуску
(running-jobs в БД чинит сам backend на старте — только показать итог)
```

## 4. UX-поток

Нативно понятные шаги, прогресс, без стектрейсов. Один линейный нарратив в
Terminal (минимум) либо нативный launcher (.command/.app, аналогичные строки).

```
Reelibra — запуск

[1/4] Проверка окружения
  Проверяю uv…        ОК
  Проверяю Node…      ОК (v22.3)
  Проверяю ffmpeg…    не найден
    → Устанавливаю через Homebrew… [прогресс brew]   ОК
  Проверяю Python 3.12…  ОК
  Зависимости backend…   ОК (без изменений)
  Зависимости frontend…  обновляю 3 пакета… ОК
  Файл .env…          ОК
    ⚠ GEMINI_API_KEY пуст — впишите ключ в Настройках после запуска

[2/4] Проверка прошлого запуска
  Порт 8000…  занят процессом 41громко (прошлый backend)
  Порт 3000…  свободен
  → Обнаружен незавершённый прошлый запуск, привожу в порядок

[3/4] Очистка
  Останавливаю старый backend…   ОК
  Освобождаю порт 8000…          ОК
  Останавливаю зависшие рендеры ffmpeg (2)…  ОК
  Удаляю временные файлы (.partial, .lock, .tmp)…  убрано 4
  Данные и БД не тронуты

[4/4] Запуск
  Backend…   http://127.0.0.1:8000/docs
  Frontend…  http://localhost:3000
  Готово. Открываю браузер…

  (прошлый запуск был прерван — 1 незавершённая задача помечена как ошибка,
   можно перезапустить её в интерфейсе)
```

Принципы вывода:
- Каждый шаг — глагол в наст. времени + результат на той же строке (`ОК`/`не
  найден`/счётчик).
- Нумерованные фазы `[N/4]` дают ощущение прогресса без прогресс-бара.
- Ошибки — человеческим языком: не `lsof: command failed`, а "Не удалось
  освободить порт 8000 — закройте процесс вручную или перезагрузите Mac".
- Предупреждения (⚠) не блокируют; ошибки (блокеры) останавливают с
  единственным понятным предложением "что сделать".
- Итог про running-jobs — мягкой строкой в конце (читается из лога backend),
  не алармом.

### Маппинг технических ошибок → человеческий текст

| Техническое | Пользователю |
|---|---|
| `uv: command not found` | "Не установлен uv. Поставить автоматически? [скачиваю…]" |
| port 3000 `EADDRINUSE` (strictPort) | "Порт 3000 занят другим приложением. Освобождаю…" / при неудаче — "закройте приложение на порту 3000" |
| `uv sync` fail (нет сети) | "Не удалось скачать зависимости — проверьте интернет" |
| `GEMINI_API_KEY` пуст | "⚠ Нет ключа Gemini — нарезка не заработает, добавьте в Настройках" |
| ffmpeg отсутствует, нет brew | "Нужен ffmpeg. Установите Homebrew (brew.sh), затем `brew install ffmpeg`" |

## 5. Что уже сделано в коде vs. что добавить launcher'у

| Возможность | Статус | Кто отвечает |
|---|---|---|
| kill stale uvicorn/vite/esbuild | есть (run.sh preflight) | run.sh — но `kill -9` сразу, без graceful |
| освобождение портов 8000/3000 | есть (run.sh) | run.sh |
| reset running-jobs → error | есть (`jobs.py:908`, lifespan) | backend на старте |
| reset Publer uploading→queued | есть (`main.py:79`) | backend на старте |
| orphan proxy-lock cleanup | есть (`proxy.py`, по mtime таймауту) | backend lazy |
| atomic write `.partial`/`.tmp` | есть (proxy/cache/font/asset) | сервисы |
| **graceful SIGTERM перед kill** | **нет** (preflight бьёт -9) | добавить launcher |
| **проверка Node-версии** | **нет** | добавить launcher |
| **проверка непустого GEMINI_API_KEY** | **нет** | добавить launcher |
| **немедленная чистка `.partial`/`.lock`/`.tmp` на старте** | **нет** | добавить launcher (опц.) |
| **PID-файл / "уже запущено"** | **нет** | добавить launcher (опц.) |
| **нумерованный UX-нарратив** | частично (echo-строки) | переоформить launcher |

Вывод: ядро самовосстановления (БД, локи, atomic-write) уже реализовано —
launcher не должен дублировать БД-логику. Его задача: (1) расширить диагностику
окружения (Node, ключ), (2) сделать чистку graceful-first, (3) обернуть всё
в понятный нумерованный UX с человеческими ошибками.
