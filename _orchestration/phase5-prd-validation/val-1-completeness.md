## 2. Непокрытые разрывы (есть в gap, нет в PRD)

### BR-06 (P0, сложность M) — КРИТИЧЕСКАЯ ДЫРА
**Gap:** assignment cancel флипает локальный статус, но пост реально уходит в соцсети. Целевое (PD3): реальный retract через **Publer DELETE**.

**Проверка PRD:** ни одно R-требование не адресует Publer-retract.
- EPIC 2 (R2.1–R2.5) трогает project_id, saved-folder, удаление legacy `/schedule`, ManualPublishButton, CTA — но НЕ отмену уже запланированной Publer-кампании.
- EPIC 3 (R3.1) — это отмена **джоба** обработки (`POST /jobs/{id}/cancel`, FL-04/MS-01), а НЕ отмена публикации. Слова "cancel" в PRD относятся только сюда.
- В метриках приёмки (стр. 73–77) Publer-retract не упомянут.

**Риск:** P0-баг остаётся в проде после реализации всего PRD — UI покажет «отменено», пост уйдёт в соцсети. Прямое нарушение принципа «ни один контрол не врёт» (цель EPIC 1) и единого честного Publer-пути (PD3).

**Рекомендация:** добавить требование в EPIC 2 (PD3-домен), напр.:
> **R2.6** (BR-06) Отмена запланированной публикации делает реальный retract через `Publer DELETE`; локальный статус флипается только при успехе API. Acceptance: отмена кампании реально снимает пост с расписания Publer, не только локальный флаг.

## 3. Остаточная dormant-проблема (PD4 реализован неполно)

### Дефолт `screencast_cursor_zoom_enabled = True` не выключен
**Контекст:** dormant cursor zoom — **toggle ВКЛ по умолчанию** (`runtime_settings.py:478`), детектор крутится на каждом screencast-джобе (полный OpenCV template-match), выход отбрасывается (`render.py:1116`, applier="deferred"). Phase 1b-fix явно постановил: «пока выключаем дефолт» (STUB-REALITY-MAP §66).

**PD4:** dormant (в т.ч. cursor zoom) — «режем/прячем, не реанимируем».

**Проверка PRD:** EPIC 9 (R9.1) удаляет только 6 orphan-модулей (object_tracker, person_cluster, match_cuts, eye_trace_continuity, transition_chooser, B-roll). Флип дефолта cursor-zoom в `False` НЕ предусмотрен ни одним R. UI этих фич уже скрыт (Phase 3), так что лжи нет — но впустую жжётся CPU, и «режем» из PD4 не доведён.

**Серьёзность:** P2 (ресурсная трата, не ложь). Должно быть явным решением, не молчаливым пропуском.

**Рекомендация:** добавить в EPIC 9 (или EPIC 8):
> **R9.2** (PD4) Выключить дефолты dormant-детекторов, которые computed-then-discard: `screencast_cursor_zoom_enabled=False` (миграция runtime_settings), чтобы не жечь CPU. Опц.: убрать осиротевшие вызовы детекторов из render.py, если безопасно.

## 5. Проверка WONTFIX-by-decision

| WONTFIX-пункт (GAP) | Явно отражён в PRD? | Обоснован? |
|---------------------|---------------------|------------|
| Auth на деструктивных эндпоинтах (single-user) | ❌ нет явной строки | Обоснован в PD1 ROADMAP («Auth НЕ добавляем — локальное single-user»). В PRD только неявно (нет auth-эпика). **Документационный пропуск.** |
| Dormant: mouth_sound, cursor zoom, deictic zoom | ⚠️ частично | UI уже скрыт (Phase 3) → лжи нет. Но cursor-zoom **default=True жжёт CPU** (§3) — это НЕ чистый wontfix, нужен флип дефолта. mouth_sound/deictic default=False — реально wontfix, ОК. |
| Orphan ~972 LOC (6 модулей) | ✅ да | R9.1 — удаление, не вывод. Корректно. |
| B-roll как продукт, persistent job queue | ❌ не упомянуты | Обоснованы в PD4 («вне скоупа честного сервиса»). B-roll частично входит в R9.1 (удаление модуля). Persistent queue — не упоминается. Допустимо (вне скоупа), но явная строка усилила бы трассируемость. |

**Документационные пропуски (низкий приоритет):** PRD не содержит явной секции «Out of scope / WONTFIX» с обоснованием (auth, persistent queue). Сейчас читателю PRD приходится сверяться с ROADMAP, чтобы понять, что эти решения осознанные, а не забытые. Рекомендуется короткая секция «WONTFIX (по решению PD1/PD4)» для трассируемости.
