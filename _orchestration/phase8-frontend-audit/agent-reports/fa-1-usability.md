> Phase 8, агент Usability. Principal UX Researcher. Эвристический разбор существующего фронта (`apps/frontend/src/`, Vite + React Router 6) под цель будущего редизайна: 2 режима — Пошаговый (новичок) / Эксперт-студия.

> Severity: **P0** — блокирует/путает большинство пользователей · **P1** — серьёзное трение · **P2** — заметное неудобство · **P3** — полировка.

## ТОП-10 проблем по severity

### U-01 — P0 — Две несовместимые дизайн-системы токенов в одном приложении
**Экран:** глобально. NavRail, SchedulerDashboard, часть HomeClient используют «студийную» тёмную палитру (`--ink-2/3`, `--gold`, `--paper`, `--mute-2`, `display-serif`, `mono`). UploadWizard, ReelCard, ExportDialog, ProjectSelector используют «светлую» систему (`--surface-raised/sunken/canvas`, `--accent-primary`, `--text-primary/secondary/muted`, `--border-default`). В `JobList.tsx`, `WizardSteps.tsx`, `UploadWizard.tsx` **обе системы смешаны в одном файле**.
**Проблема:** визуальная бессвязность — левый рейл выглядит как другой продукт, чем форма справа. «Режим монтажа» в визарде даже хардкодит `bg-white`/`text-stone-900`/`text-stone-500` (строки 736–771), третий стиль. Пользователь не считывает единый бренд; кнопки/карточки в разных частях выглядят по-разному → подрывает доверие.
**Рекоменд. для редизайна:** один источник токенов. Слить в единую палитру (предположительно «студийную» как brand-ведущую), вычистить `stone-*` хардкоды. Это предпосылка для обоих режимов — без единой системы «Пошаговый» и «Эксперт» будут выглядеть как два разных приложения.

### U-02 — P0 — Settings продублированы в двух навигациях, нет единой точки «Настройки»
**Экран:** NavRail + SettingsSubNav. NavRail выносит 6 из 8 подразделов settings как top-level пункты («Профили», «Модели», «Субтитры», «Пост-продакшн», «Промпты», «Производительность»). Внутри `/settings/*` — отдельный `SettingsSubNav` с **всеми 8** (+ «Фирменные стили», «Обслуживание», скрытые из рейла). Нет пункта «Настройки» как такового.
**Проблема:** 9-пунктовый рейл на 80% состоит из настроек обработки — новичок видит «Промпты», «Производительность», «Пост-продакшн» наравне со «Студией» и теряется, что из этого основной поток, а что тонкая крутилка. Два пункта (brand, maintenance) достижимы только через под-nav → их не найти из рейла. Активное состояние двоится (рейл и subnav подсвечивают одно и то же).
**Рекоменд.:** свернуть все настройки под один пункт «Настройки» (или «Студия настроек» для эксперт-режима). Рейл новичка = 3–4 пункта (Создать / Мои нарезки / Шедулер / Настройки). Эксперт-режим может раскрывать подразделы.

### U-03 — P0 — Главный «визард» — это не визард, а одна длинная форма со всеми опциями сразу
**Экран:** `UploadWizard.tsx` (главная). 6 пронумерованных Step + аккордеон «Дополнительно» + «Режим монтажа» + ComposerStrategy — всё на одном скролле, без next/back, без принудительной последовательности. Кнопка сабмита одна, в самом низу.
**Проблема:** максимальная когнитивная нагрузка в самой первой точке контакта. Новичок, который хочет «загрузить видео → получить рилсы», сразу видит соотношение сторон, пресеты субтитров, per-job override чекбоксы (интро/аутро/зум/loudnorm/ч-б), split-screen превью, доп-промпт на 8000 знаков, транскрайбер, LLM-провайдер, fit-mode, proxy 1080p, force reingest, composer strategy. Нумерация «Step 1–6» создаёт ложное ожидание мастера, которого нет.
**Рекоменд.:** это и есть ядро задачи редизайна. **Пошаговый режим:** настоящий мастер (file → формат → авто-режим → старт), всё остальное скрыто за «умными» дефолтами. **Эксперт-студия:** текущая плоская форма, но честно помеченная как экспертная, без обманной нумерации шагов. Auto-режим (`AutoConfigSummary`) — сильная сторона, его надо делать дефолтным мостом для новичка.

### U-04 — P1 — Нет отмены идущей обработки
**Экран:** прогресс-карточка в UploadWizard (строки 863–902), JobDetail. SSE показывает стадию+%, но **кнопки «Отменить» нет** (бэк `POST .../cancel` существует, `JobStatus.cancelled` обрабатывается в SSE-хуке). Запустив нарезку 2-часового видео, пользователь не может её остановить из UI — только перезагрузка/ожидание.
**Рекоменд.:** кнопка «Отменить обработку» рядом с прогресс-баром (и в JobDetail). Подтверждение через confirm. Критично для длинных джобов.

### U-05 — P1 — Экспорт обещает разное качество, но это заглушка
**Экран:** `ExportDialog.tsx`. Пресеты TikTok (6 Мбит), Reels (5 Мбит), Shorts (8 Мбит) с битрейт/LUFS-лейблами, но дисклеймер (строки 147–148): «MVP: возвращает ссылку на существующий MP4 с метаданными пресета. Full transcode по bitrate — в следующей итерации».
**Проблема:** UI создаёт ложное ожидание перекодирования под платформу. Лейблы «6 Мбит / 8 Мбит» декоративны — пользователь выбирает «качество», которого не получает. Честный дисклеймер мелким текстом не спасает: сам выбор пресета бессмысленный.
**Рекоменд.:** до реализации transcode — убрать битрейт-лейблы и иллюзию выбора, оставить одну кнопку «Скачать MP4». Либо реализовать transcode. Полу-правда хуже отсутствия фичи.

### U-06 — P1 — Viral-score выглядит как вердикт движка, но это клиентская эвристика
**Экран:** `ReelCard` (ScoreBadge, крупная цифра /100 с цветовой градацией A–C), ReelGrid (фильтр «Топ»), ClipDetail. Считается на клиенте через `viralScore.ts` из длины+ритма (`computeViralScore`), не из пайплайна.
**Проблема:** крупное цветное кольцо «87/100» — самый заметный элемент карточки, читается как объективная оценка ИИ-движка. Дисклеймер «клиентская эвристика по длине и ритму, не оценка движка нарезки» спрятан в раскрывашке breakdown. Пользователь будет отбраковывать хорошие рилсы по фейковому числу.
**Рекоменд.:** либо подключить реальный per-reel score из бэка, либо радикально снизить визуальный вес (не доминирующая цифра, а скромный бейдж «~ритм») и вынести дисклеймер на поверхность. Для эксперт-режима — оставить как фильтр-помощник, не как вердикт.

### U-07 — P2 — Проекты: создание есть, навигация в папку и обратная связь хромают
**Экран:** UploadWizard (селект «Проект» появляется только если `projects.length > 0`), `/projects`, `ProjectFolderPage` (lazy). Селект проекта показывается только когда проекты уже созданы — новичок без проектов его не видит и не понимает, что папки существуют. «Сохранить в папку» (`saved/<folder>`) даёт toast, но путь к просмотру сохранённого неочевиден.
**Рекоменд.:** в редизайне — связать проект с потоком явно (выбор/создание проекта как опциональный шаг мастера), и дать видимый экран содержимого проекта/папки с CTA «куда сохранили».

### U-08 — P2 — Переход к результату после `done` легко пропустить
**Экран:** UploadWizard прогресс-карточка → мелкая ссылка «Открыть детали →» (строки 893–900). Частично смягчено в HomeClient: при отсутствии активных джобов показывается крупный CTA «Смотреть рилсы» (строки 72–92). Но если пользователь остался на скролле визарда, единственный путь дальше — мелкий текст в карточке.
**Рекоменд.:** на `done` прямо в прогресс-карточке делать крупную primary-кнопку «Смотреть рилсы», а не текстовую ссылку. Консистентно с HomeClient-CTA.

### U-09 — P2 — Inconsistency интерактивных паттернов (кнопки, подтверждения, тоггл-группы)
**Экран:** глобально.
- **Кнопки:** где-то `.btn .btn-primary` (SchedulerDashboard, HomeClient CTA), где-то полностью инлайновые Tailwind-классы primary (UploadWizard submit, строки 791) — нет единого Button-компонента.
- **Подтверждения:** удаление кампании — нативный `confirm()` (SchedulerDashboard 76), удаление рилса — нативный `window.confirm()` (ReelCard 331), нет единой модалки подтверждения.
- **Тоггл-группы:** «сегментные» переключатели (aspect, reel-count auto/custom, split-screen preset/on/off) каждый раз свёрстаны заново инлайн с чуть разными классами вместо общего компонента.
**Рекоменд.:** ввести общие примитивы (Button, ConfirmDialog, SegmentedControl) — обязательное условие для двух режимов с консистентным look&feel.

### U-10 — P3 — Онбординга и empty-state-гайдов почти нет
**Экран:** глобально. Единственная подсказка новичку — inline-ошибка «задай GEMINI_API_KEY» (UploadWizard 140–148). Нет welcome-тура, нет «создай первый проект → загрузи → …», у `/scheduler` неплохой empty-state («мастер за четыре шага…»), но он единичный.
**Рекоменд.:** для пошагового режима — лёгкий first-run онбординг (1–2 экрана) + осмысленные empty-states на главной и в проектах. Не туры-оверлеи, а контекстные подсказки в пустых зонах.

## Прочие наблюдения (ниже топ-10)

- **NavRail mono-коды (DSH/PRJ/SCH/PRF…)** — стильно, но для новичка непрозрачно; коды не несут смысла, занимают место. В пошаговом режиме убрать.
- **Аккордеон «Дополнительно»** в визарде смешивает реально продвинутое (force reingest, proxy) с тем, что новичку всё же нужно (язык исходника) — граница «основное/доп» проведена не по уровню пользователя.
- **«Режим нейросети»** в визарде — это не контрол, а текст-отсылка в `/settings/performance`. Раздражающий редирект внутри формы: показывает опцию, но управлять ей нельзя здесь.
- **Cross-context badge** (ReelCard 190–225) — хороший паттерн контекстного предупреждения, стоит тиражировать.
- **`/settings/maintenance` и `/settings/brand`** недостижимы из глобального рейла — только через под-nav (см. U-02).

## ВЕРДИКТ: новичок vs эксперт

**Эксперту — удобно (почти).** Вся мощь на поверхности: один скролл-визард с 6 шагами + аккордеон даёт прямой доступ к транскрайберу, провайдеру, fit-mode, proxy, composer strategy, per-job override постпродакшна; `/settings/performance` с ~25 группами параметров; редактор 12 промптов; ASS-субтитры. Для человека, который знает, что такое loudnorm и J/L-cut, это эффективная панель. Трение эксперта — только в consistency (U-01, U-09) и в фейковых обещаниях (U-05, U-06), которые раздражают именно знающего пользователя.

**Новичку — НЕ удобно.** Продукт спроектирован «от автора инструмента»: первый экран вываливает все настройки сразу (U-03), глобальный рейл на 80% состоит из экспертных крутилок (U-02), нет онбординга (U-10), а ключевые «успокаивающие» элементы (score, экспорт-пресеты) вводят в заблуждение (U-05, U-06). Единственное, что спасает новичка сегодня — **Auto-режим**: он реально берёт решения на себя и вместе с живым SSE-прогрессом и пост-`done` CTA образует связный коридор «загрузил → робот всё сделал → смотрю рилсы». Этот коридор и есть зародыш «Пошагового режима».

**Итог для редизайна.** Разделение на 2 режима — правильный ответ на текущую дисфункцию. Костяк уже есть:
1. **Пошаговый** строить вокруг Auto-режима + настоящего мастера (file → формат → «робот соберёт» → прогресс → результат), всё экспертное скрыть.
2. **Эксперт-студия** = текущая плоская форма + полный settings, но честно помеченная, без обманной нумерации шагов.
3. **Блокеры до старта режимов:** единая дизайн-система токенов (U-01) и единая точка «Настройки» в IA (U-02) — без них оба режима будут визуально и навигационно рассыпаться.
4. **Снять полу-правду:** экспорт-заглушка (U-05) и фейк-score (U-06) бьют по доверию в обоих режимах.

Цельного путешествия «от загрузки до опубликованного поста» пока нет, но публикация теперь связана лучше, чем фиксировала карта Phase 3 (ManualPublishButton и единый Publer-шедулер живы). Главная рана — вход: первый экран рассчитан на эксперта, а приходит новичок.
