# FA-5 — Аудит здоровья кода и производительности фронтенда

**Скоуп:** `apps/frontend/src` (143 .ts/.tsx, ~21 000 LOC). Стек: React 19 + react-router-dom 7 (data router) + Vite 7 (SWC) + Tailwind 4. Без UI-библиотек, без стейт-менеджеров.
**Цель:** оценка перед редизайном + 2 режима.

---

## Вердикт о готовности к редизайну: СРЕДНЕ (ближе к лёгкому)

Архитектура на удивление здоровая для самописного фронта. Логика и презентация **разделены частично, но систематически** — есть чёткий слой data (loaders + api/), чистый стейт-хук (`useSettingsSave`), токенизированный CSS (89 из 118 компонентных файлов используют `var(--*)`). Это значит, что новый визуал накладывается в основном через `globals.css` + JSX-разметку, без переписывания бизнес-логики.

**Что делает редизайн «средним», а не «лёгким»:**
1. **Презентация и логика живут в одних файлах.** Нет слоя UI-примитивов (Button/Card/Input/Modal/Badge). Каждый компонент сам верстает Tailwind-классы инлайн (CampaignDetailClient — 89 `className=`). Перевёрстка = редактирование сотен инлайн-классов в десятках крупных компонентов, а не замена нескольких примитивов.
2. **Дуальный режим (2 темы) сейчас невозможен без работы.** `globals.css` имеет ровно один `:root` с `color-scheme: dark` — захардкоженная тёмная тема. Нет ни `[data-theme]`, ни `.light/.dark`, ни `prefers-color-scheme`, ни переключателя. Токены есть (хорошо), но они одноуровневые: семантические `--text-primary` ссылаются на физические `--paper/--ink`, и второго набора физических значений под светлую тему не существует.

**Хорошая новость:** оба пункта решаются аддитивно, без рефактора логики. См. рекомендации по дуальному режиму ниже.

## 1. Архитектура компонентов

Структура чёткая и предсказуемая:
- `pages/` — тонкие route-компоненты: `loader()` (data-fetch) + дефолтный экспорт, который рендерит `*Client`-компонент с `initial*` пропсами. Это калька с Next.js App Router (Server Component → Client Component), сохранённая после порта на Vite.
- `components/` — фиче-сгруппированы (`job/`, `scheduler/`, `settings/`, `upload/`, `dashboard/`, `shell/`, `projects/`).
- `lib/api/` — доменно-разбитый API-клиент.
- `lib/`, `hooks/` — утилиты и переиспользуемая логика.

**Сильные стороны:** консистентный page→loader→Client паттерн, фиче-нарезка, выделенные `settings-shared/` примитивы (Group/NumberRow/SliderRow/SelectRow/SwitchRow) и `performance-groups/` (28 мелких групп вместо одного God-компонента — следы Phase 8 декомпозиции).

### Проблемы

|----|-----|-----|--------|
| FA5-A1 | HIGH | весь `components/` | **Нет слоя UI-примитивов.** Кнопки/карточки/инпуты/бейджи/модалки верстаются инлайн в каждом компоненте. Дублирование разметки и главный множитель стоимости редизайна. → Перед редизайном выделить `components/ui/` (Button, Card, Input, Select, Modal, Badge, Field) и прогнать крупные компоненты через них. ## 2. Управление состоянием

**Паттерн:** локальный `useState` + router loaders как источник серверных данных. **Ноль глобального стейта** — нет Context, Redux, Zustand, Jotai, useReducer (подтверждено grep'ом: 0 совпадений). Серверные данные приходят через `loader()` и `initial*` пропсы, дальше живут в локальном стейте.

Это осознанно простой подход и для приложения такого размера он **в основном уместен** — нет «хаоса стейта».

| ID | Sev | Где | Проблема / Рекомендация |
|----|-----|-----|--------|
| FA5-S1 | MED | settings-клиенты + крупные клиенты | **Умеренный prop-drilling.** Серверные данные прокидываются page → Client → под-компоненты через `initial*` пропсы и `setValues`/сеттеры вниз по дереву. Терпимо сейчас, но при редизайне с общим shell/theme усилится. → Завести узкий Context **только** для cross-cutting concern: тема + (после редизайна) текущий режим. Не тащить туда доменные данные — loaders справляются. |
| FA5-S2 | LOW | `useSettingsSave.ts` | `dirty` через `JSON.stringify(values) !== JSON.stringify(initial)` на каждый рендер. Для крупных settings-объектов — лишняя сериализация. → Передавать `equals` явно либо мемоизировать сериализацию. Низкий приоритет. |
| FA5-S3 | LOW | `useWizardState.ts`, локальные хуки | Хорошо: где стейт сложный — вынесен в кастом-хук (wizard, settings-save). Логика отделена от верстки. Это и есть причина, почему редизайн «средний», а не «тяжёлый». |

## 3. API-слой

**Организация — образцовая.** `lib/api/core.ts` (единый `request<T>` + `ApiError` + `resolveUrl`) и доменные модули (jobs/scheduler/settings/subtitle/post_production/projects/proxies), агрегированные в `api`. Типобезопасность сильная: generic `request<T>`, экспортируемые типы ответов, типизированный `ApiError` со `status`/`detail`.

| ID | Sev | Где | Проблема / Рекомендация |
|----|-----|-----|--------|
| FA5-P1 | MED | `pages/*.tsx` loaders | **Тихое глотание ошибок:** почти все loaders делают `.catch(() => null/[])` (HomePage — 7 параллельных fetch, все с fallback). UI деградирует молча — пользователь не отличает «нет данных» от «бэкенд упал» (кроме HomePage, где есть явный экран). → Различать сетевую ошибку и пустой ответ; показывать toast/баннер деградации. Не блокер редизайна, но UX-долг. |
| FA5-P2 | LOW | `components/job/WaveformBar.tsx` | Единственный инлайн `fetch(` в обход api-слоя. → Перенести в `lib/api`. |
| FA5-P3 | LOW | `lib/api/core.ts` `resolveUrl` | Функция-no-op (возвращает path как есть). Историческое наследие Next.js. → Можно инлайнить, косметика. |

## 4. Производительность

**Главная проблема — монолитный бандл.** Подтверждённые цифры из `dist/`:
- `index-*.js` = **705 789 B raw / 191 411 B gzip** — один чанк.
- Code-splitting есть **только** для `ProjectFolderPage` (lazy, 2.9 KB).
- CSS = 78 KB raw / 16.6 KB gzip (Tailwind 4, приемлемо).
- Шрифты self-hosted wo? (хорошо), но грузятся все 3 семейства × множество subset'ов (inter latin-ext 85 KB, cyrillic-ext 26 KB и т.д.) — суммарно ~0.5 MB шрифтов.

| ID | Sev | Где | Проблема / Рекомендация |
|----|-----|-----|--------|
| FA5-PF1 | HIGH | `router.tsx` | **Нет code-splitting по роутам.** Все 23 страницы + их Client-компоненты статически импортированы в `router.tsx` → один 705 KB чанк на первый заход на любую страницу. → `lazy(() => import())` для всех route-элементов (паттерн уже применён к ProjectFolderPage). Loaders можно оставить eager или тоже динамить. Ожидаемый эффект: initial bundle 191 KB gzip → ~60-90 KB. Самый дешёвый и крупный выигрыш. |
| FA5-PF2 | MED | весь дерево компонентов | **Нулевая мемоизация.** Ноль `React.memo`, ~суммарно мало `useMemo/useCallback`. Тяжёлые списки (`ReelGrid` 338, `JobList` 327, `ScheduleTimeline` 417) ре-рендерят все дети при любом изменении родительского стейта. SSE-апдейты (`useJobSse` шлёт `lastEvent` каждый прогресс-тик) триггерят каскад. → После выделения примитивов обернуть строки списков (`ReelCard`, job-rows) в `memo`; стабилизировать колбэки `useCallback`. |
| FA5-PF3 | MED | `index.html` / шрифты | 3 шрифтовых семейства со всеми subset'ами (~0.5 MB woff2). → Грузить только нужные subset'ы (latin + cyrillic), `font-display: swap`, preload только критичных. |
| FA5-PF4 | LOW | `vite.config.ts` | `sourcemap: true` в проде (index.js.map = 2.5 MB). Если деплоятся map'ы — отключить для прода или ограничить `hidden`. |
| FA5-PF5 | LOW | `useJobSse` | Каждый SSE-эвент → `setLastEvent` → ре-рендер всего поддерева подписчика. С PF2 не критично. Можно throttle прогресса. |

## 5. Готовность к редизайну (детально)

**Разделение логики и презентации: ЧАСТИЧНОЕ, но систематическое.**
- ✅ **Data-логика отделена:** loaders + `lib/api` + кастом-хуки (`useSettingsSave`, `useWizardState`) — это переживёт редизайн без изменений.
- ✅ **Токенизация CSS:** 89/118 файлов используют `var(--*)`, семантические токены (`--text-primary`, `--surface-card`, `--accent-primary`, `--danger`) маппятся на физическую палитру. Смена палитры = правка `globals.css`, не компонентов.
- ⚠️ **Верстка НЕ отделена:** инлайн Tailwind в каждом компоненте, нет UI-kit. Перевёрстка = ручная работа по многим файлам.
- ⚠️ **Хардкод hex** в ~9 файлах, но это **семантические данные**, не UI-chrome: градиент viral-score (`ReelCard` #4ade80…#f87171), дефолты бренд-кита, fallback-цвет проекта. Низкий риск для темизации — их можно оставить или вынести в токены опционально.

### Наложение 2 режимов поверх текущей архитектуры — насколько легко

**Если «2 режима» = light/dark тема:** СРЕДНЕ-ЛЕГКО, чисто аддитивно.
1. Текущий `:root` → переименовать в `:root, [data-theme="dark"]`.
2. Добавить `[data-theme="light"]` со вторым набором **физических** значений (`--ink/--paper/--line` и т.д.). Семантический слой (`--text-primary` → `--paper`) переписывать не нужно — он уже абстрагирован.
3. Узкий ThemeContext + переключатель + `localStorage`, ставит `data-theme` на `<html>`.
4. Убрать `color-scheme: dark` хардкод → делать его частью темы.
   Риск: те ~9 хардкод-hex и инлайновые `var(--danger)/10` opacity — проверить контраст в светлой теме.

**Если «2 режима» = два функциональных режима UI** (напр. simple/advanced, как `narrative_mode` на бэке): СРЕДНЕ — потребует условного рендера в компонентах, но стейт-инфраструктура (Context/проп) тривиальна; основная стоимость снова в отсутствии UI-примитивов.

### Что переиспользуемо / что снести
- **Переиспользовать как есть:** весь `lib/` (api, sse, router-compat, viralScore, video-thumbnail), `hooks/`, loaders, `settings-shared/` и `performance-groups/` примитивы, доменная типизация.
- **Переверстать (логику сохранить):** все `*Client.tsx`, `pages/`, shell (`AppShell/NavRail/TopBar`), карточки/списки.
- **Снести/рефакторить:** дубль-фасад `api.ts` (после миграции импортов), no-op `resolveUrl`, инлайн `fetch` в `WaveformBar`.

## 6. Технический долг (сводно)

1. **`router-compat.ts`** — эмуляция `next/navigation` (usePathname/useRouter/notFound) поверх react-router. Работает, но это переходное наследие порта Next.js→Vite. Места вызова используют `useRouter().push()` вместо нативного `useNavigate`. Не срочно, но это «скрытый слой», который путает новых разработчиков. Зафиксировать как осознанный compat либо постепенно мигрировать на нативный react-router.
2. **page→Client дублирование** — каждая страница = 2 файла (тонкий page + Client) из-за Next.js-наследия. В SPA это лишний бойлерплейт (можно слить loader прямо в компонент), но менять рискованно и не приоритетно.
3. **Молчаливая деградация loaders** (FA5-P1) — UX-долг.
4. **Sourcemap в проде** (FA5-PF4).

## TOP-долги (приоритет для подготовки к редизайну)

1. **FA5-PF1 (HIGH)** — route-level code-splitting через `lazy()`. Один чанк 705 KB / 191 KB gzip → дешёвый кратный выигрыш. Сделать ДО редизайна.
2. **FA5-A1 (HIGH)** — выделить `components/ui/` примитивы перед перевёрсткой. Это главный множитель стоимости редизайна И предусловие для эффективной мемоизации (FA5-PF2).
3. **Дуальная тема (FA5 §5)** — добавить `[data-theme="light"]` физический набор + ThemeContext. Аддитивно, не трогает логику.
4. **FA5-A2 (MED)** — декомпозировать 5 God-компонентов (CampaignDetailClient 878 первым).
5. **FA5-PF2 (MED)** — мемоизация списков после выделения примитивов.

## Резюме одной строкой

Здоровый, типобезопасный, токенизированный фронт с чистым data-слоем — но без UI-примитивов, без code-splitting и без light-темы. Редизайн **средней** сложности: бизнес-логика переживёт его нетронутой, вся работа уходит в перевёрстку (нужен UI-kit) и аддитивную дуальную тему. Сначала — split бандла + UI-примитивы, потом редизайн.
