## 0. Идеология режима (soul + проф)

Эксперт-студия — это пульт монтажёра, а не форма регистрации. Принцип из soul: **уважение к эксперту** (вся мощь под рукой, ничего не спрятано за «упрощением») при **«подсказке напротив каждого контрола»** (плотность не означает «угадай сам»).

Три закона студии:

1. **Плотность без хаоса.** Сетка из панелей даёт высокую информационную плотность, но дышит за счёт брендовых интервалов (Sumi-карточки на Kuro-фоне, тонкие Hai-разделители, прямые углы).
2. **100% подсказок.** Ни одна галочка / ползунок / селект / кнопка не существует без `<Tooltip>`. Это не «бест-практис», это gate: компонент без подсказки не проходит ревью.
3. **Честность вместо фикций.** Бэкенд содержит декоративные ручки (tier `pro` физически = Flash-Lite; export не перекодирует — устарело после Phase 6; dormant-zoom жжёт CPU и выбрасывает результат). Студия не прячет их, но честно маркирует через систему бейджей (см. §5). Эксперт имеет право знать, что ручка декоративна.

## 2. TOOLTIP-СИСТЕМА — `<Tooltip>` / `<ControlHint>`

Ядро режима. Переиспользует паттерн обязательного `hint`-пропа из settings-примитивов (FRONTEND-AUDIT: «settings ~100% через обязательный `hint`-проп примитивов»). В Эксперт-студии этот паттерн расширяется на **все** контролы, не только settings.

### 2.1 Анатомия подсказки — формула 3 частей

Каждая подсказка строится по фиксированному шаблону (это и есть гарантия качества и единообразия):

```
┌─ Tooltip ─────────────────────────────┐
│  ЧТО ДЕЛАЕТ   (1 строка, по-русски)    │  ← обязательно
│  ЭФФЕКТ       (на скорость/качество/    │  ← обязательно
│               стоимость/кадр)           │
│  РЕКОМЕНДАЦИЯ  «Оставьте X для Y»        │  ← обязательно
│  [бейдж честности, если применимо]      │  ← по ситуации (§5)
└────────────────────────────────────────┘
```

Пример (ползунок «Целевое число рилсов»):
> **Сколько рилсов нарезать из видео.** Больше число — дольше анализ и рендер, ниже средняя «вирусность» каждого. Для часового подкаста оставьте 12–18.

### 2.2 Два слоя подсказки

Плотность требует двух уровней, чтобы не загромождать экран:

| Слой | Триггер | Содержимое | Где |
|---|---|---|---|
| **Inline-hint** | всегда видим | первая строка («что делает»), `Manrope 13px Kasumi #8A8278` | прямо под/справа от контрола, серым |
| **Full-tooltip** | hover / focus / tap-на-`(i)` | все 3 части + бейдж | всплывающая Sumi-карточка |

Inline-hint обеспечивает «подсказку напротив каждого контрола» без действий пользователя. Full-tooltip даёт глубину по требованию. Так достигается плотность БЕЗ хаоса: глаз видит короткую подсказку всегда, длинную — когда нужно.

### 2.3 Спецификация компонента `<Tooltip>`

```
<Tooltip
  what="Сколько рилсов нарезать из видео"        // обязательный, string
  effect="Больше число — дольше анализ и рендер"  // обязательный, string
  advise="Для часового подкаста оставьте 12–18"   // обязательный, string
  badge?="decorative" | "off-default" | "cpu-heavy" | "opt-in" | "experimental"
  side?="left"        // студия: подсказки слева от P3 (контролы прижаты вправо)
>
  <слот-триггер />   // оборачивает любой контрол или иконку (i)
</Tooltip>
```

**Триггеры (мультимодальность):**
- **Hover** (мышь): задержка 150ms перед показом, мгновенное скрытие при уходе.
- **Focus** (клавиатура): показ при `:focus-visible` на контроле или его `(i)`-кнопке. Критично для доступности — эксперт ходит по форме с Tab.
- **Tap** (тач): на тач-устройствах hover нет → у каждого контрола рядом видимая иконка `(i)` 44×44px (touch-target из чеклиста), tap по ней показывает/прячет тултип. Решает VD-03 из аудита (действия не должны быть hover-only на тач).

**Позиционирование:**
- В P3 контролы прижаты к правому краю узкой панели → тултип открывается `side="left"`, чтобы не уезжать за экран.
- Авто-флип, если не помещается (правый/верхний/нижний фолбэк). Никогда не перекрывает сам контрол.
- На mobile (P3 = full-screen sheet) тултип = разворачивающийся блок под контролом (accordion-style), не floating — floating на узком экране режет текст.

**Стиль (брендбук):**
```
.tooltip-card {
  background: #1A1A1A;            /* Sumi */
  border: 1px solid #2A2A2A;      /* Hai */
  border-left: 2px solid #C9A84C; /* Kinzoku — акцентный «корешок» */
  border-radius: 0;               /* прямые углы */
  padding: 12px 16px;
  max-width: 280px;
  box-shadow: 0 8px 24px rgba(0,0,0,0.55);  /* тёплая тень, не серая */
}
.tooltip-what   { font: 600 14px Manrope; color: #F0E6D2; }
.tooltip-effect { font: 400 13px Manrope; color: #8A8278; margin-top: 6px; }
.tooltip-advise { font: 400 13px Manrope; color: #F0E6D2; margin-top: 6px; }
.tooltip-advise::before { content: "→ "; color: #B87333; }
.tooltip-badge  { font: 9px 'Press Start 2P'; margin-top: 10px; }
```
- Появление: fade-in + translate-y(4px), 0.15s ease-out (в рамках брендбука «до 1с», без bounce/elastic — запрещены).
- Никаких эмодзи внутри. Бейдж — текст-чип (см. §5).

### 2.4 Как достигается 100% покрытие (механика, не обещание)

Покрытие — это инвариант, обеспеченный архитектурой примитивов, а не дисциплиной разработчика:

1. **Обязательный проп.** Все UI-примитивы студии (`<ToggleRow>`, `<SliderRow>`, `<SelectRow>`, `<NumberRow>`, `<TextRow>`, `<ActionButton>`) принимают `what/effect/advise` как **required** поля. Пропустить нельзя — типобезопасность не даст собрать компонент без подсказки (расширение паттерна `hint`-проп из FRONTEND-AUDIT на ВСЕ контролы).
2. **Один источник подсказок.** Реестр `controlHints` (по ключу настройки/действия) — словарь `{ key → {what, effect, advise, badge} }`. Все ~81 ручка + действия P4 описаны централизованно. Примитив достаёт по своему `name`/`id`.
3. **Gate-валидатор.** При сборке проходит проверка: каждый отрендеренный примитив имеет непустые `what/effect/advise`. Контрол без записи в реестре → fail сборки. Это переносит «100% подсказок» из категории «надеемся» в категорию «гарантировано».
4. **Кнопки тоже.** «Запустить», «Сбросить группу», per-reel «export/like/опубликовать», «Сохранить пресет» — оборачиваются в `<ActionButton>` с тем же обязательным триплетом. Для кнопок `advise` = что произойдёт и обратимо ли («Удаление рилса необратимо. Останется в saved/, если копировали»).

Итог: контролов без подсказки физически не существует в DOM студии.

## 4. ЧТО ЭКСПЕРТ ВИДИТ ДОПОЛНИТЕЛЬНО (vs Пошаговый)

Пошаговый (визард) показывает ~8–10 ключевых ручек с безопасными дефолтами. Эксперт-студия раскрывает всё остальное:

**Видимо только в Эксперте:**
- Все off-default DSP-трансформы (группа 3), reducer ensemble / multi-arc / cross-chunk (группа 1).
- Редактор 12 промптов pipeline + свой системный промпт.
- Сырые tier/провайдер/модель + проба `/health`.
- Прямой редактор `.ass`-субтитров конкретного рилса.
- Обслуживание прокси, purge-уровни, force re-ingest.
- 9-region vision-overlay и face-bbox прямо на превью.
- Honesty-бейджи (decorative/dormant/broken) — Пошаговый их просто не показывает, скрывая фиктивные ручки.

**Быстрые действия (header / ⌘K палитра):**
- `⌘K` командная палитра: поиск любой из ~81 настройки по имени → прыжок к контролу в P3.
- «Запустить» (primary CTA, Kinzoku-заливка) — `POST /jobs` с текущей конфигурацией.
- «Сбросить группу к дефолтам» в шапке каждой группы аккордеона.
- «Анализ аудио» / «Подсказка профиля» — one-shot действия без полного прогона.
- Фильтр настроек (🔍 над P3): мгновенно сужает аккордеон по подстроке.

**Пресеты конфигурации:**
- Сохранить текущий набор всех 8 групп как именованный пресет (поверх существующих perf/vision/subtitle/post-prod store).
- Применить пресет → разом заполняет студию. Подсказка пресета описывает «для какого контента» (подкаст / вебинар / talking-head / скринкаст).
- Auto-config (`PATCH /jobs/{id}/auto-config`) — кнопка «Подобрать автоматически» применяет LLM-рекомендацию к job.options; «Сбросить в ручной» (`DELETE .../auto-config`).

## 5. СИСТЕМА БЕЙДЖЕЙ ЧЕСТНОСТИ

Бэкенд содержит ручки, которые не делают того, что обещают (BACKEND-MAP §6, §2.7). Студия не прячет их, а маркирует. Бейдж = текст-чип `Press Start 2P 9px`, ховер открывает full-tooltip с честным объяснением.

| Бейдж | Цвет | Значение | Пример |
|---|---|---|---|
| `OPT-IN` | Kasumi #8A8278 | выключено по умолчанию, безопасно включить | весь Vision |
| `OFF-DEFAULT` | Kasumi | выключено, влияет на скорость/стоимость | reducer ensemble |
| `CPU-HEAVY` | Dō #B87333 | жжёт CPU/время | face-tracking |
| `DORMANT` | Dō #B87333 | считается, но результат выбрасывается | mouth-sound removal, cursor-zoom |
| `DECORATIVE` | Chi #8B2500 (точечно) | ручка ничего не меняет | tier pro/flash |
| `BROKEN` | Chi #8B2500 | помечено автором как сломанное | chaptered mode |
| `PARTIAL` | Chi #8B2500 | работает частично | assignment cancel |
| `DESTRUCTIVE` | Chi #8B2500 | необратимое действие | purge=nuke |

Chi (#8B2500) используется строго точечно (1–3% макета, правило палитры) — только на действительно опасных/фиктивных ручках. Это и есть «капля крови воина» из брендбука в функциональном виде.

## 6. SPACING / ТИПОГРАФИКА / АДАПТИВ (engineering-инвариант)

Из брендбука + чеклиста frontend-design (8px-grid, плавная прогрессия, mobile-first).

**Section/group padding (P3 аккордеон):**
```
Заголовок группы:        py-4  (16px)
Между контролами:        gap-3 (12px)  ← плотно, это студия
Между группами:          py-6 md:py-8  (24→32px)  ← воздух
Поле контрола:           p-3.5 (14px) по брендбуку
Tooltip-карточка:        p-3 (12px 16px)
```
Никаких скачков >24px. Контейнер P3 — свой `overflow-y: auto`, остальной layout без фикс. px-высот (кроме шапки 56px и ленты P4 ~280px — намеренные).

**Типографика:**
- Заголовки групп — `Noto Serif JP Bold` (display, реально отличается от body).
- Тех-лейблы, бейджи, мета — `JetBrains Mono` / `Press Start 2P` (бейджи).
- Подсказки, тело — `Manrope`.
- Размеры подсказок 13–14px — сознательно крупнее аудитного «text-[11px]» (VD-05 чинит мелкий текст; «17px-философия» брендбука уважается в подсказках).

**Адаптив (брендбук breakpoints):**
- `>1024px` — все 4 панели одновременно (студийный пульт).
- `768–1024` — P1 сворачивается в иконки, P3 — drawer по кнопке «Настройки», P2/P4 в стек.
- `<768` — P3 = full-screen sheet (тултипы становятся accordion-блоками, не floating), P1 = bottom-tab, галерея P4 — `grid-cols-1 sm:grid-cols-2`. Touch-таргеты `(i)` и действия ≥44px.

**Доступность:**
- Tab-навигация по всем контролам, `:focus-visible` → Kinzoku-бордер + показ full-tooltip.
- `prefers-reduced-motion` → отключает fade/translate тултипов.
- ARIA: каждый контрол `aria-describedby` → id своей подсказки (подсказка не только визуальная).

## 7. ПЕРЕКЛЮЧЕНИЕ В ПОШАГОВЫЙ

- Тумблер в header: `[⚙ Эксперт ▾ | Пошаговый]` — управляется общим `UiModeContext` (FRONTEND-AUDIT, фундамент п.3).
- Состояние конфигурации **сохраняется** при переключении: настройки, выставленные в Эксперте, не сбрасываются — Пошаговый просто показывает их подмножество.
- При переходе Эксперт → Пошаговый: если включены ручки, которых нет в визарде (напр. face-tracking, свой промпт) — ненавязчивый бейдж в визарде «+N экспертных настроек активно» с ссылкой назад. Честность без блокировки.
- При переходе Пошаговый → Эксперт: студия открывается на группе, соответствующей последнему шагу визарда (контекст не теряется).
- Выбор режима персистится (localStorage) — оператор, выбравший студию, при следующем заходе попадает в неё.

## ИТОГ (для оркестратора)

**Layout студии:** 4 панели на Kuro-фоне — P1 навигатор-якоря (200px sticky), P2 источник+таймлайн+SSE stage-bar (fluid центр), P3 аккордеон из 8 групп со всеми ~81 ручкой (420px, свой скролл), P4 лента результатов-рилсов (280px снизу). Плотно внутри групп (gap-3), воздух между (py-8). Прямые углы, Sumi-карточки, один Kinzoku-акцент ≤30%, адаптив сворачивает P1→иконки и P3→drawer/sheet.

**Tooltip-система:** компонент `<Tooltip what/effect/advise/badge>` с двухслойной подачей — всегда-видимый inline-hint (1 строка серым под контролом) + full-tooltip по hover/focus/tap-`(i)` (3 части: что делает / эффект / рекомендация, Sumi-карточка с Kinzoku-корешком, side=left, авто-флип, на mobile = accordion). Мультимодальность (мышь/клавиатура/тач) + ARIA `aria-describedby`.

**100% покрытие достигнуто механически, не дисциплиной:** (1) все UI-примитивы делают `what/effect/advise` обязательными required-пропами — собрать контрол без подсказки нельзя типобезопасно; (2) единый реестр `controlHints` описывает все ~81 ручку + действия P4 централизованно; (3) gate-валидатор сборки падает, если хоть один отрендеренный примитив без подсказки; (4) кнопки и per-reel действия оборачиваются тем же примитивом. Это расширение паттерна обязательного `hint`-пропа из FRONTEND-AUDIT на ВСЕ контролы, а не только settings.

**Бонус — честность:** 8 honesty-бейджей (opt-in/off-default/cpu-heavy/dormant/decorative/broken/partial/destructive) маркируют фиктивные и опасные ручки из BACKEND-MAP §6, используя Chi #8B2500 точечно по правилу палитры. Эксперт видит правду, Пошаговый эти ручки скрывает.
